# ADR-002: 画面認識にOCR＋YOLOのハイブリッド構成を採用する

## ステータス
承認済み

## 日付
2026-02-24

## 文脈

ポケモン対戦画面には構造化された情報（HP数値・ポケモン名・技名・スコア）が多い。これらをすべてVision APIで処理すると呼び出し頻度が増えコストが増大する。一方で状態異常アイコンやボール数などはOCRでは取得しにくい。

## 決定

数値・テキスト情報はEasyOCR、アイコン類はYOLOv8でローカル処理する。Vision APIは大きな状況変化の補完用途に限定する。

## 対象と取得方法の対応表

| 情報 | 取得方法 |
|------|---------|
| HP数値・バー | EasyOCR |
| ポケモン名 | EasyOCR |
| 技名 | EasyOCR |
| 状態異常アイコン | YOLOv8 |
| 残りポケモン数（ボール） | YOLOv8 |
| 天気・フィールド効果 | EasyOCR / YOLOv8 |
| ターン切替の検知 | OpenCV 差分検出（cv2.absdiff） |

## 理由

- ローカル処理は無料でAPIコストを大幅に削減できる
- ポケモン対戦画面はUIが固定されておりOCRの精度が出やすい
- YOLOv8はVRAM消費が少なく（0.5〜1GB）他コンポーネントと共存可能
- Vision APIの呼び出しをイベント時のみに絞ることでコストを制御できる

## 却下した選択肢

| 選択肢 | 却下理由 |
|--------|---------|
| 全量Vision API | 毎フレーム呼ぶとコスト過大 |
| 全量ローカルVision LLM | VRAM不足・3Dモデルと共存不可 |

## リスクと対策

- ゲームのUIが更新されるとYOLOの再学習が必要になる可能性がある
- OCRがポケモン名を誤認識した場合はポケモン名辞書で後補正する

## 結果

APIコストを抑えながら必要な情報を網羅的に取得できる。Sprint 1ではまずOCR＋差分検出のみで試作し、Sprint 4でYOLOを追加する段階的な実装とする。
