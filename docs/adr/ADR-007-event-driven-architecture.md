# ADR-007: イベント駆動アーキテクチャによる処理トリガーを採用する

## ステータス
承認済み

## 日付
2026-02-25

## 文脈

毎フレームすべてのコンポーネントを動かすとCPU/GPU負荷が高く、APIコストも増大する。画面に変化がないターン中は処理をスキップし、イベント発生時にのみ各コンポーネントを起動するイベント駆動設計が必要。

「どの変化をイベントと見なすか」「イベントごとにどのコンポーネントを呼ぶか」の定義がなければ実装が発散するため、本ADRで明示的に定義する。

## 決定

以下のトリガー条件とコンポーネント呼び出しルールを採用する。

## イベント定義とトリガー条件

| イベント種別 | 検知方法 | 判定条件 |
|------------|---------|---------|
| ターン切替 | OpenCV差分検出 | フレーム差分が閾値（TBD）を超えた場合 |
| ポケモン交代 | EasyOCR | ポケモン名テキストが変化した場合 |
| 技使用 | EasyOCR | 技名テキストが新たに出現した場合 |
| ポケモン気絶 | YOLOv8 / OCR | HP数値が0になった場合またはボール数が減少した場合 |
| 状態異常付与 | YOLOv8 | 状態異常アイコンが新たに検出された場合 |
| 天気・フィールド変化 | EasyOCR / YOLOv8 | 天気アイコン・テキストが変化した場合 |

## イベント発生時のコンポーネント呼び出しルール

```
イベント検知（OpenCV / EasyOCR / YOLOv8）
　↓
[必須] 状況テキスト構築（OCR/YOLO結果をまとめてテキスト化）
　↓
[条件付き] Bedrock Claude Haiku Vision呼び出し
    → 大きな状況変化イベントの場合のみ（ターン切替・交代・気絶）
    → 技使用・状態異常は状況テキストのみで実況生成を試みる
　↓
[必須] Phi-3 mini（Ollama）で実況文生成
    → Bedrock分析結果 + OCR/YOLO情報 + 直前の実況履歴をプロンプトに渡す
　↓
[必須] VOICEVOX APIで音声生成
　↓
[必須] 音声再生 + VMC Protocolで口パク同期
　↓
[必須] S3に実況ログ・スクリーンショット保存
```

## Bedrockを呼ぶイベントと呼ばないイベントの分類

| イベント | Bedrock呼び出し | 理由 |
|---------|--------------|------|
| ターン切替 | あり | 全体状況を把握したい最重要イベント |
| ポケモン交代 | あり | 対戦の流れが大きく変わるため |
| ポケモン気絶 | あり | 試合の決定的な場面 |
| 技使用 | なし | OCR取得した技名・ダメージで十分 |
| 状態異常付与 | なし | YOLOでアイコン検出すれば情報足りる |
| 天気・フィールド変化 | なし | テキストで検知できるため |

## ポーリング間隔

- 画面キャプチャ間隔: **1秒ごと**（mssで取得）
- OpenCV差分検出: キャプチャごとに実行（CPU処理で軽量）
- OCR処理: 差分検出でイベントありと判定した場合のみ実行

## 処理の優先度と遅延目標

| ステップ | 目標遅延 |
|---------|---------|
| イベント検知〜Bedrock呼び出し | < 0.5秒 |
| Bedrock Vision分析 | < 1秒 |
| Phi-3 mini実況文生成 | < 1秒 |
| VOICEVOX音声生成 | < 0.5秒 |
| **合計（イベント〜音声再生）** | **< 3秒** |

## 却下した選択肢

| 選択肢 | 却下理由 |
|--------|---------|
| 毎フレーム全処理 | CPU/GPU負荷過大・APIコスト爆発 |
| 固定インターバル呼び出し（例：毎5秒） | イベントのタイミングと合わないため実況が不自然になる |
| キューイング（Celery等） | Sprint初期段階では過剰。Sprint 4以降で検討 |

## 今後の拡張

- Sprint 5以降で実況の連続性を高めるため、直前N件の実況テキストをPhi-3のプロンプトに含める
- 同一イベントの連続検知を防ぐデバウンス処理（例：同じポケモン名が連続2回以上検出された場合は1回だけトリガー）を追加する

## 結果

不要なAPI呼び出しを排除しコストを制御しながら、3秒以内の実況遅延を実現できる。各コンポーネントの責務とトリガー条件が明確になり実装が発散しない。
